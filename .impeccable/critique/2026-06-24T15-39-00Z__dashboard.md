## Anti-Patterns Verdict

The OKLCH palette with hue 38 (Court Copper), the deliberate dark-on-void-black theme, the `ElapsedTimer` as a domain-specific insight, and the thorough `aria-label` coverage all signal real design intent. ## Overall Impression

The shell is right. Dark theme, copper accent, real-time timers, thorough a11y labels — this reads like a tool built by someone who thought about the use case. The score (26/40) reflects not visual problems but **interaction gaps**: a silent court assignment (the primary action), a doubly-ambiguous result picker in doubles, and unlabeled navigation on the device that matters most. Fix those three and the score climbs into the low 30s. The emotional register is also flat — the peak moment (assigning a matchup to a court) has no closure signal at all.

## What's Working

1. **Confirmation discipline.** Every destructive action — queue remove, void match, session close — uses a two-step confirm/cancel flow with a Cancel escape. Consistent in intent even where visual treatment varies. This is the right call for a high-stakes, no-undo context.

2. **`ElapsedTimer` on active courts.** Live elapsed time per court card is a genuine domain insight. Organizers need to know when a match is running long; this answers that without requiring any navigation. Not a default pattern — it was thought through.

3. **Aria labels are specific and correct.** `aria-label={\`Move ${entry.player.name} up\`}`, `aria-label={\`End match on Court ${number}\`}` — contextual, not generic. Most AI output leaves these as "button" or omits them entirely. This signals care.

## Priority Issues

**[P0] Doubles result picker identifies side by one first name only**
- **What**: In `CourtCard.tsx` lines 106–114, the "who won?" grid renders three buttons using `firstName(activeMatch.sideA[0]?.name ?? "Side A")` / Draw / `firstName(activeMatch.sideB[0]?.name ?? "Side B")`. In a doubles match with two players per side, only one player's first name labels that side's button. If both sides have an "Alex" (common in amateur leagues), or if the organizer can't recall who's on which side mid-match, they misrecord the result.
- **Why it matters**: Match results drive leaderboard and ratings. A wrong result is silent and hard to detect after the fact — it damages player trust in the system.
- **Fix**: Show both player first names stacked on the side button (`Alice / Bob` on Side A), or use "Side A / Draw / Side B" as primary labels with small player names as subtext. Either eliminates the ambiguity.
- **Suggested command**: `/impeccable polish`

**[P1] No feedback on court assignment — the primary action is silent**
- **What**: `handleCardAssign` in `DashboardClient.tsx` line 92 removes the assigned card and inserts a new empty card with no feedback — no toast, no animation, no "Court 2 — Assigned" confirmation. The organizer sees the card disappear and a new empty card appear.
- **Why it matters**: This is the highest-stakes action in the app. In a courtside environment with distractions, a silent state change creates "did that work?" anxiety and encourages repeat taps. It's also the worst moment to lose confidence in the tool.
- **Fix**: A 300ms toast ("Assigned → Court 2") or a brief card transform (check mark flash before removal) establishes closure. Either takes < 30 lines.
- **Suggested command**: `/impeccable animate`

**[P1] Tablet navigation has no labels — sidebar is icon-only on the primary device**
- **What**: `Sidebar.tsx` renders at `md:` (768px+) with icon-only navigation and CSS `:hover` tooltips (`group-hover:opacity-100`). `BottomBar.tsx` is `md:hidden`. A 10–12" tablet — the stated primary device — gets the icon sidebar without the labeled bottom bar. Hover tooltips are a mouse pattern; tablets don't hover.
- **Why it matters**: Navigation requires recall on the device that needs recognition most. An organizer who hasn't used the app in a week will tap icons and guess.
- **Fix**: At `md:` breakpoint, either render the BottomBar instead of (or alongside) the icon sidebar, or add unconditional text labels below each sidebar icon. The sidebar is already 64px wide — even 48px labels fit if the icon shrinks slightly.
- **Suggested command**: `/impeccable adapt`

**[P2] No button variant system — every button is hand-crafted**
- **What**: Across `CourtCard.tsx`, `PlanningCard.tsx`, `QueueMatchupSection.tsx`, and `CourtsSection.tsx`, every button has individually specified border-radius (`rounded-sm` vs `rounded-md`), padding, font size, and border treatment. "New Match" is `rounded-md`, "End" is `rounded-sm`, "Assign" is `rounded-sm`, "Add" (courts) is `rounded-md`.
- **Why it matters**: Visual inconsistency reads as unfinished. It also makes global changes (e.g., "make all buttons pill-shaped") require touching every file.
- **Fix**: Extract a `Button` component with 3 variants — `primary` (copper fill), `ghost` (transparent, border optional), `danger` (error tint). All use `rounded-md`. Apply across the dashboard in one pass.
- **Suggested command**: `/impeccable polish`

**[P2] Planning card drag affordance is invisible on touch**
- **What**: `PlanningCard.tsx` line 107 sets `cursor-grab` on draggable cards. The drag-to-court assignment is described as the "primary method on tablet+" in the spec, but `cursor-grab` is a CSS property — invisible on touch devices. There is no grip icon, no visual drag handle, no hint text.
- **Why it matters**: If organizers don't discover drag, they fall back to the Assign dropdown (2 extra taps per assignment). The UX centerpiece is effectively hidden on the primary device.
- **Fix**: Add a `GripVertical` icon to the card header, always visible. On `touch` devices the cursor doesn't exist — the icon is the only affordance.
- **Suggested command**: `/impeccable polish`

## Persona Red Flags

**Alex (Power User / Head Organizer)**: Managing 4 courts + 12 players. Primary concern: speed and accuracy.
- The queue reorder (up/down arrows) is fine for 4 entries but painfully slow for 10+. No drag-to-reorder in the queue. At 10 entries, moving a player from position 8 to position 1 requires 7 taps.
- No keyboard shortcuts. Tab/Enter navigation not verified as complete.
- The planning card resuggest button exists but is a no-op (no handler wired) — a power user will click it immediately and get no response.

**Sam (Accessibility-Dependent User)**: Screen reader + keyboard-only.
- `aria-label` coverage is strong — a genuine strength.
- The `draggable` planning card has no keyboard alternative for assignment. Drag-to-court can only be done with a pointer. The Assign dropdown exists as a fallback, but it's a `<div>`-relative dropdown — keyboard focus behavior after open needs verification.
- The `QueueMatchupSection` inline remove confirmation renders a `<Check>` and `<X>` button with no aria-live announcement. A screen reader user won't know the confirmation appeared.

**Casey (Distracted Mobile User)**: Phone, one-handed, courtside. Frequent interruptions.
- Queue management on mobile: the right column (with waitlist + planning cards) appears ABOVE the courts on mobile (`order-1`). This is correct — the organizer needs the queue first. But the planning cards horizontal scroll is 85vw per card on mobile. With `scroll-snap-type: x mandatory`, the first card is fully visible but the second card requires a deliberate swipe. No scroll indicator (dots, arrows) tells the organizer there are more cards.
- The 44px minimum touch target is respected on most buttons, but the planning card dismiss `X` button is `p-1.5` with an 11px icon — the hit area is approximately 30×30px. Below the 44px threshold.
- Bottom bar on mobile shows: Dashboard, Sessions, Matches, Players, Courts. The active tab is Dashboard, which is where queue management lives. This is correct — but the tab label "Dashboard" doesn't communicate "this is where you manage matches." An organizer interrupted mid-session returning to the app may not know which tab contains their active work.

## Minor Observations

- `CourtsSection.tsx` line 56: "2 courts available" footer text is redundant — StatusBadge on each card already communicates availability. Remove or replace with something actionable.
- `QueueMatchupSection.tsx` line 114: "Upcoming" with `uppercase tracking-wider` at 11px is the one AI-eyebrow tell in an otherwise clean codebase. Replace with a plain `h3` at normal case, or remove the label entirely (the section is self-evident).
- `QueueRow` position number: `w-4` only comfortably fits single digits. At position 10+, the number will be cramped. Use `w-6 min-w-[1.5rem]`.
- `PlanningCard.tsx` line 175: the fallback player `{ id: "", name: "—", skillLevel: "C", status: "ACTIVE" }` is a code smell. Render an empty slot div instead of a fake player object.
- `CourtCard.tsx` drop target: `bg-primary/5` at 5% opacity is nearly invisible on the `--color-bg` surface (`0.09 0 0`). Increase to `/12` or `/15` to make the drop zone legible.
- `SessionHeader.tsx`: `{session.playerCount}p` ("12p") is ambiguous shorthand. Use a `Users` icon or "12 players" at wider breakpoints.
- The resuggest button in `PlanningCard.tsx` has no handler wired — it's a visual affordance with no behavior. This will confuse power users immediately.
- `NoSessionState` in `page.tsx`: "Start Session" button has no `onClick` — currently a non-functional CTA.

## Questions to Consider

1. **Why do planning cards and the queue coexist in the same panel without showing their relationship?** The queue defines who plays next — planning cards should visually derive from the queue (e.g., highlight the queue entries used in each card). Right now they feel like two independent lists sharing a container.
2. **What happens when all 4 courts are occupied?** The planning section is still visible and still shows matchup suggestions despite no court being assignable. Does showing "upcoming" matchups during full occupancy serve the organizer, or create a false sense of urgency that distracts from watching active matches?
3. **Is horizontal scroll the right pattern for planning cards on a touch device?** Under time pressure, an organizer may not scroll to discover card 2 or 3. Would a vertical stack of 2 cards with a "+1 more" collapse serve the primary workflow better?
