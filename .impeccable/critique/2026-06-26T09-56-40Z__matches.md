## Anti-Patterns Verdict

**LLM assessment**: PASS with brand-character gaps. The component avoids the core AI traps — no side-stripe borders, no gradient text, no hero-metric template, no identical card grid. The context-aware Win/Loss badge flip is genuinely non-generic. However it reads as "technically proficient dark dashboard" rather than "The Director's Bench." Copper (`oklch(0.71 0.17 38)`) is absent from data rows and filter interactions. Active filter pills use fully neutral chrome (`bg-surface-elevated text-ink border border-border`), identical to a hovered-but-not-pressed state. The badge system's `side_a` vs `side_b` distinction (copper vs accent blue) creates directional contrast that is semantically ambiguous — does blue mean "opponent's result" or "secondary outcome"?

**Deterministic scan**: Clean — exit code 0, zero findings. No absolute bans, no slop patterns detected in markup.

## Overall Impression

A solid architectural foundation with one safety gap that blocks courtside trust: void is permanent with no undo, and the confirmation UI is too quiet to stop a misfire on a phone in a gym. The context-aware badge flip is the page's most valuable feature but is completely hidden behind the search interaction — most organizers will never discover it. The "Director's Bench" premium feel is present in restraint but absent in presence; the copper brand color does no work on this page.

## What's Working

**1. Context-aware badge system.** `getBadgeVariant` flipping between `side_a/side_b` → `win/loss` based on the active search player is the most sophisticated interaction in the codebase. It elevates search from "filter" to "personal stats lookup" without a separate view — exactly the kind of earned density the brand brief calls for.

**2. Void flow architecture.** `confirmingId` state replacing a single row's action area (not a modal) is the right pattern for tablet-first courtside use. It keeps context visible, is cancellable, and the trigger button carries a full `aria-label` with the matchup string.

**3. Token system discipline.** `font-mono tabular-nums` applied correctly to the count badge and time values. Semantic OKLCH aliases throughout. Z-index scale used properly (`var(--z-sticky)`, `var(--z-toast)`). The global `prefers-reduced-motion` block handles animation correctly.

## Priority Issues

**[P0] No undo on void — irreversible destructive action with no recovery path**
- **Why it matters**: A courtside organizer standing in a loud gym with a phone, running sessions in real-time, will occasionally tap the wrong row or confirm accidentally. There is no way back. The mock state makes this survivable, but production data loss from a misfire is a trust-destroying event. The "Director's Bench" brand requires operational confidence — and right now void fires a one-way door.
- **Fix**: Extend `showToast` to accept `onUndo?: () => void`. Store a `lastVoided` ref (`useRef<MatchRecord | null>`). When void confirms, save the pre-void match to the ref, then show `"Match voided — Undo"` where "Undo" is an inline button that restores the match from the ref. The undo window matches the toast timeout (3s). After 3s the window closes and the void is permanent.
- **Suggested command**: `/impeccable polish`

**[P1] Confirming row has no visual anchor — user can lose track on dense lists**
- **Why it matters**: When `confirmingId` is set, only the action area changes (two tiny text buttons appear). The row itself is identical to all others. On a list of 9+ rows while standing, a user who pauses before confirming can lose track of which match is about to be voided — and accidentally void the wrong one.
- **Fix**: Apply `ring-1 ring-inset ring-error/30 bg-error/5` to the root `div` of a row where `isConfirming === true`. This gives a quiet red halo to the active row without being alarming. The background tint makes the row unmistakably distinct.
- **Suggested command**: `/impeccable polish`

**[P1] Void confirm tap targets below spec (36px vs 44px minimum)**
- **Why it matters**: Both "Cancel" and "Void" in the confirm state have `min-h-[36px]` — 8px below the PRODUCT.md minimum, and 12px below the 48px target for primary session actions. The destructive "Void" confirm is a primary session action. Courtside, standing, one-handed on a phone — the miss rate for a 36px target is materially higher.
- **Fix**: Raise both confirm-state buttons to `min-h-[44px]` minimum, `min-h-[48px]` for the destructive "Void". Add `px-3` to widen the horizontal tap zone. Additionally, add `aria-label` attributes to both confirm buttons (e.g. `"Confirm void"` / `"Cancel void"`) so screen readers and assistive technology identify them with context.
- **Suggested command**: `/impeccable polish`

**[P2] "Side A" / "Side B" badge labels expose internal enum vocabulary**
- **Why it matters**: When no player is searched, the result badge reads "Side A" or "Side B" — the raw data field values, not a human outcome. An organizer reviewing the log 20 minutes after a match must mentally reconstruct "Side A = Alice & Bob's team" from memory. This is the Working Memory Violation the cognitive load checklist flags.
- **Fix**: Two options. (a) In the no-search state, replace "Side A / Side B" with the winning side's first names: `formatSide(match.result === "SIDE_A" ? match.sideA : match.sideB)` — max ~12 chars on a doubled-badge. (b) Simpler: use a direction arrow (`← Side A` / `Side B →`) or a neutral label ("Result") with a copper/blue color cue only, reserving text labels for the searched context. Option (a) is more informative but requires a wider badge.
- **Suggested command**: `/impeccable polish`

**[P2] Flat list with no temporal grouping — orientation breaks at 10+ matches**
- **Why it matters**: A 3-hour session with 9+ matches is an undifferentiated flat list. The organizer must read every `time` value to mentally orient. Compare to Vercel's deployment list or GitHub's commit history — temporal grouping (hour boundaries, "Earlier today") is standard for activity logs. The DESIGN.md explicitly calls out "Earned density — every relevant piece of information visible without scrolling during a session."
- **Fix**: Group rows under a sticky section label keyed by hour or time block (e.g. "11:00 AM", "12:00 PM"). Implement as a derived array in `useMemo` that interleaves `{ type: "group-header", label: string }` entries with match records, rendered as `<div role="rowgroup">` separators. The time values on each row can be simplified to minutes-only once grouped.
- **Suggested command**: `/impeccable layout`

## Persona Red Flags

**Alex (Impatient Power User)**
- No keyboard navigation between filter pills — must mouse-click each one.
- Result filters are single-select; can't view Wins + Draws simultaneously.
- No sort control (fixed descending `startedAt`; no court, player, or type sort).
- On 30+ match sessions, flat list with no grouping is a scrolling slog with no orientation anchors.
- Badge flip behavior (search to see Win/Loss) requires accidental discovery — no tooltip or hint that this exists.

**Sam (Accessibility-Dependent)**
- `min-h-[36px]` on confirm-state buttons fails WCAG 2.5.5 (44×44px target).
- No focus management when confirm state appears — keyboard/switch access users must navigate manually to Cancel/Void.
- Confirm buttons lack `aria-label` with match context; only the trigger button has it.
- "Draw" and "Voided" badges: `text-muted` (`oklch(0.62)`) on `bg-surface-elevated` (`oklch(0.18)`) ~4.8:1 — meets AA but falls short of the stated ≥7:1 courtside body text target.

**The Organizer (courtside, standing, bright gym, tablet or phone)**
- Void confirm buttons too small and too quiet for one-handed use in bright ambient light.
- No haptic or strong visual feedback when void fires — `opacity-55` dim is subtle on a backlit tablet in a gym.
- Badge flip behavior (search for Win/Loss) is undiscoverable; the primary question after each match ("who won?") is not immediately legible in unfiltered view.
- Checking a player's match count requires: type name in search → count rows → mentally filter voided → remember the number. Pure working memory burden across a 3-hour session.

## Minor Observations

- `opacity-55` on voided rows is a non-standard Tailwind value — verify it is extended in `tailwind.config.ts` or it silently falls back to opacity 1.
- Active filter pill uses `bg-surface-elevated text-ink border border-border` — identical to a hover state. Should use `text-primary` or `border-primary/50` to signal selection, consistent with PlayersView skill chips which use `ring-1 ring-primary/50` for active.
- The `Activity` icon in the empty state is generic. At this brand touch-point, a shuttlecock SVG or custom icon would do brand work.
- `passesResultFilter` has a dead `return true` fallback at the bottom — not a bug, but a code-smell that a future maintainer may misread as a missing branch.
- Search input `sm:w-[240px] flex-shrink-0` leaves visible dead space in the filter bar on tablet when no filters are active and "Clear" is hidden.
- `h1` at `text-base font-semibold` is identical weight to row content — the page heading lacks authority, especially compared to the DESIGN.md's `headline` spec (600, ~1.5rem).

## Questions to Consider

- What if the result badge defaulted to the winning side's first names instead of "Side A" — would that feel more like a scoreboard and less like a database?
- Does the void action need a 3-second undo, or should it be a full "Are you sure?" with a typed confirmation (given that voiding changes recorded history)?
- What would happen if the filter pills used the copper active state — would the page feel more alive, or too loud?
